## 二、分层解释（自底向上）

### Layer 1：LLM + Embedding（发动机）

| 组件 | 技术 | 为什么 |
|------|------|--------|
| 主力 LLM | DeepSeek V3（deepseek-chat） | OpenAI-compatible API，便宜且强，切换 cost 为 0 |
| 复杂推理 | DeepSeek R1（deepseek-reasoner） | 用于 verify_answer 这种严谨多维度判断 |
| Embedding | fastembed BAAI/bge-small-en-v1.5 | 本地 384-dim，免费，不消耗 API token |

ChatOpenAI 封装了三者（`chat()` / `chat_structured()` / `chat_stream()`），切换 provider 只改 base_url。

### Layer 2：数据与工具（轮子）

- **Qdrant**：存储论文的向量（Dense 语义 + Sparse BM25 关键词），用 Hybrid RRF 融合检索
- **PostgreSQL**：业务数据（求职记录状态机、研究笔记、ingestion 日志）
- **LangGraph Store**：用户长期记忆（偏好、收藏、计划摘要）
- **Sandbox**：subprocess + AST 静态检查，安全执行 AI 生成的代码
- **8 个 Tools**：search_papers / fetch_paper_details / explain_concept / generate_backtest_code / create_research_plan / generate_interview_questions / update_user_memory / track_application

### Layer 3：Agent 编排（方向盘 + 变速箱）

**Parent Graph** 负责：
1. Router 用 LLM 判断用户意图（research / codegen / planning / interview，支持跨域）
2. Send API 把任务分发到一个或多个子图（跨域问题触发并行 fan-out）
3. 合并结果，格式化最终输出

**四个 Subgraph** 各自独立执行：

| 子图 | 流程 | 特点 |
|------|------|------|
| Research | parse → search → verify → retry? ↻ → format | 检索验证闭环，优雅降级 |
| CodeGen | parse → **interrupt** ⏸️ → generate → sandbox → retry? ↻ | 人机协作 |
| Planning | parse → generate → format | 纯线性 |
| Interview | parse → search → generate → format | 线性，按 JD 生成面试题 |

**AgentState** 是所有节点共享的 dict，带 reducer 处理并发写冲突。

### Layer 4：API + UI（方向盘）

- **FastAPI**：`POST /chat`（发送消息）、`POST /resume`（从 interrupt 恢复）
- **Streamlit**：浏览器聊天界面
- **Docker Compose**：一键启动 Qdrant + PostgreSQL + API + UI

### Layer 5：可观测性（仪表盘）

- **LangSmith**：自动 trace 每次图执行的完整过程（节点延时、token 消耗、错误堆栈）
- **Checkpointer**：自动持久化每个节点的 State 快照，支持对话恢复和崩溃恢复

## 三、请求的完整生命线（以 CodeGen 为例）

```
用户在 Streamlit 输入: "写一个RSI<30买入、RSI>70卖出的回测"

1. Streamlit → POST /chat {user_id, thread_id, message}

2. FastAPI 调用 graph.invoke({"messages": [...], "user_id": "xxx"}, config)

3. Parent Graph 执行:
   a. Router (LLM) → {primary_mode: "codegen", confidence: 0.92}
   b. route_to_subgraphs → [Send("codegen", {})]

4. CodeGen Subgraph 执行:
   a. parse_strategy (LLM) → {name:"RSI", rsi_period:14, buy:30, sell:70, ...}
   b. confirm_with_user → interrupt() ⏸️ 暂停！

5. API 返回: {"status": "interrupt", "interrupt": {"interrupt_id":"...", "strategy_spec":{...}}}

6. Streamlit 显示参数表单，"用户把 RSI 周期改成 21，点确认"

7. Streamlit → POST /resume {thread_id, interrupt_id, edited_spec:{rsi_period:21,...}}

8. CodeGen Subgraph 继续:
   c. generate_code (LLM) → 生成 backtrader Python 代码
   d. execute_sandbox → subprocess 运行，检查语法/执行/输出
   e. format_success → 返回代码+运行结果

9. API 返回: {"status": "ok", "response": "Strategy: RSI...\nGenerated code:\n...\nSandbox output:\n..."}
```

## 四、英文介绍（三种长度，直接可用）

### 15-Second Elevator Pitch

> I built QuantMind, a multi-agent AI copilot for quantitative finance research and job hunting. It uses LangGraph to orchestrate four specialized agents — Research, Code Generation, Planning, and Interview — with a Hybrid RAG system on Qdrant. The CodeGen agent has a human-in-the-loop interrupt so users can review strategy parameters before code execution. It's fully containerized with Docker Compose, has 100 passing tests, and uses LangSmith for full-stack tracing.

### 1-Minute Project Summary

> I built a quantitative finance AI copilot called QuantMind. The tech stack is LangGraph for agent orchestration, Qdrant for hybrid vector search, DeepSeek V3 as the LLM, and FastAPI plus Streamlit for the interface.
>
> The core architecture is a parent graph with four subgraphs — Research, CodeGen, Planning, and Interview. The parent router uses an LLM to classify user intent, then fans out to one or more subgraphs in parallel for cross-domain queries.
>
> The RAG system combines dense embeddings from BAAI/bge-small with sparse BM25 vectors using Qdrant's reciprocal rank fusion. I run embeddings locally with fastembed — no API costs. The retrieval has a confidence check, a verification step with four quality criteria, a retry loop, and graceful decline if evidence is insufficient.
>
> The CodeGen agent has a human-in-the-loop interrupt: the LLM first parses the trading strategy, then pauses for the user to confirm or edit the parameters before generating and sandbox-executing the code.
>
> For evaluation, I built a 120-question benchmark with three evaluators — LLM-as-judge, code execution, and citation checking. I used LangSmith traces to diagnose and fix eight production issues discovered during evaluation.

### Detailed Walkthrough (for technical interview)

> **Architecture**: The system has five layers:
>
> **Layer 1 — LLM & Embedding**: I use DeepSeek V3 through LangChain's ChatOpenAI class since DeepSeek has an OpenAI-compatible API. For embeddings, I use fastembed locally with BAAI/bge-small-en-v1.5 — 384 dimensions, free, no API calls needed.
>
> **Layer 2 — Data & Tools**: Papers are ingested from ArXiv, chunked, and stored in Qdrant with both dense and sparse vectors. Hybrid search uses RRF to merge results. Business data goes to PostgreSQL via SQLAlchemy. I also have eight LangChain tools that the agents call — paper search, concept explanation, backtest code generation, interview question generation, job application tracking, and user memory management.
>
> **Layer 3 — Agent Orchestration**: This is where LangGraph does the heavy lifting. I have a parent graph with a router node that uses structured LLM output to classify user intent into one of four modes. It can also fan out to multiple subgraphs for cross-domain queries using LangGraph's Send API.
>
> Each subgraph is independently compiled and injected into the parent as a node. The Research subgraph is the most complex — it parses the query, runs hybrid search, checks confidence, decides whether to expand search depth, synthesizes an answer, verifies it against four criteria using an LLM judge, and retries once before degrading gracefully.
>
> The CodeGen subgraph uses LangGraph's interrupt mechanism for human-in-the-loop. After the LLM parses the strategy specification, it pauses. The API returns an interrupt payload to the frontend. The user edits the parameters and hits confirm. The backend resumes with `Command(resume={interrupt_id: edited_spec})` — critically, you have to use the interrupt_id mapping, not a bare value. The node re-executes from the top, so the interrupt call must come before any irreversible side effects.
>
> State management is three-tier: the Checkpointer for thread-level conversation state (automatic persistence), the LangGraph Store for cross-thread user memory (preferences, bookmarks), and PostgreSQL for structured business data. I used reducers on State channels that get written concurrently — without them, parallel fan-out causes `InvalidUpdateError`.
>
> **Layer 4 — API & UI**: FastAPI exposes `/chat` and `/resume`, the Streamlit UI provides a chat interface, and everything runs in Docker Compose.
>
> **Layer 5 — Observability**: LangSmith traces the full execution graph. This was critical during evaluation — I discovered eight production issues through trace inspection, including a 14-second retrieval latency caused by sparse seed data, a CodeEvaluator crash from a misused `del` statement, and a parallel write conflict on a State field that needed a reducer.
